# Observability & Operational Guardrails

This document defines the operational practices needed to keep V2 reliable and auditable, especially for:
- LLM evaluation calls (Gemini)
- strict JSON contract validation
- immutable event persistence
- voice streaming transport (two-way audio)

It aligns with the “enterprise choke point” principle in `FULL_FINAL_BLUEPRINT.md`.

## Correlation and traceability

### Correlation IDs (required)
Every request that triggers evaluation must carry a correlation id end-to-end:
- `requestId`: generated by the frontend (optional) or backend
- `correlationId`: stable id used in logs across:
  - API request
  - LLM call
  - event append

Suggested fields to log:
- `tenantId`
- `sessionId`
- `nodeId`
- `turnId`
- `clientSubmissionId`
- `eventId` (when created)
- `modelId` / `promptVersion` / `scoringVersion`

### Structured logging (no string concatenation)
Log objects:
- `logger.info({ msg: "...", tenantId, sessionId, nodeId, turnId, ... })`
- Include sanitized params only (no raw secrets or PII).

## Metrics and monitoring (what to measure)

### API-level metrics
- request count by endpoint:
  - `/api/missions/start`
  - `/api/missions/decision`
  - `/api/missions/mentor`
- latency percentiles (p50/p95/p99)
- error rates by status code
- error code distribution:
  - `EVAL_JSON_INVALID`
  - `SESSION_TERMINAL`
  - `INVALID_NODE`
  - etc.

### LLM evaluation metrics
- success rate of strict JSON parsing
- validation failure rate
- time spent in:
  - prompt assembly
  - LLM call
  - parsing/validation
  - profile update & event persistence
- token/cost estimates (if provider exposes usage)

### Event lake metrics
- events append success/failure
- event create latency
- any “write then respond” anomalies

### Voice streaming metrics (frontend + optional backend relay)
Track transport reliability:
- mic capture started time
- first AI audio chunk latency
- average time between chunks
- interruption frequency
- transcript finalization rate
- turn discard rate (due to interruption/empty transcripts)

## Timeout budgets and retries (bounded)

### LLM call timeouts
Define hard timeouts:
- connect timeout
- response timeout

Behavior on timeout:
- do not update profiles
- write `EVALUATION_LLM_ERROR`
- return a safe API error

### Retry policy
- Retry only bounded and only for safe categories:
  - transient network errors
  - provider timeouts
- For strict JSON validation:
  - allow at most one “repair and re-output” attempt
  - never loop indefinitely

## Circuit breakers and degradation
If evaluation failures exceed a threshold:
- degrade gracefully:
  - show a UI message like “Evaluation temporarily unavailable”
- stop expensive retries
- keep mission UI responsive (do not block indefinitely)

## Cost controls
- cap max retries
- cap max concurrent evaluation jobs per tenant (if implementing job queues)
- prefer:
  - smaller models for social engine flavor (if applicable)
  - strict contract models for scoring only

## Guardrails: what must never happen
- Never store secrets in events or client logs.
- Never log raw user PII in structured logs.
- Never update existing `/events` documents.
- Never advance `MissionState` unless:
  - idempotency checks pass
  - state machine validations pass
  - and evaluation output passes strict validation (for open input)

## “Evidence mode” for audit/debug (admin-only)
When admins inspect audit trails:
- allow viewing:
  - evaluation event type and reasons
  - scoring versions
- keep safe boundaries:
  - hide or redact any sensitive input excerpts
